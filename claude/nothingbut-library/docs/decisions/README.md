# 决策日志 (Architecture Decision Records)

本目录记录项目中的关键技术和设计决策，采用 ADR (Architecture Decision Record) 格式。

## 什么是 ADR？

ADR 是一种轻量级的文档格式，用于记录软件开发过程中的重要决策：
- **背景**: 为什么需要做这个决策？
- **决策**: 选择了什么方案？
- **考虑的方案**: 还考虑了哪些备选方案？
- **后果**: 这个决策带来什么影响？

## 为什么需要 ADR？

### 问题：上下文丢失

在多会话开发中，常见问题：
```
Session 1: 需求澄清，用户确认 UI 原型 ✅
    ↓
Session 2: 设计阶段，基于文字描述设计 ⚠️
    ↓ (原型信息丢失)
Session 3: 开发阶段，与原型有偏差 ❌
    ↓
Session 4: 测试阶段，重新澄清需求 ⏰
```

### 解决方案：决策追溯

ADR 记录"为什么"，而非仅记录"是什么"：
```
设计文档: "使用 2 栏布局" (结果)
ADR-001: "为什么选择 2 栏而非 3 栏？用户确认了哪个原型？" (过程)
```

## 何时创建 ADR？

### 必须创建 ADR 的情况

1. **技术选型**
   - 选择框架、库、工具
   - 例：Tauri vs Electron vs Wails

2. **架构决策**
   - 系统架构设计
   - 数据流设计
   - 例：MVC vs MVVM vs Clean Architecture

3. **UI/UX 重大变更**
   - 布局结构调整
   - 交互流程优化
   - 例：3 栏改为 2 栏

4. **技术规范**
   - 编码规范
   - 命名约定
   - 例：参数使用 camelCase

5. **性能/安全策略**
   - 权限配置
   - 数据加密
   - 例：显式声明 Tauri 权限

6. **用户反馈驱动的变更**
   - UX 优化
   - 流程简化
   - 例：导入流程改为立即解析

### 不需要创建 ADR 的情况

- 纯 bug 修复（无决策变更）
- 代码重构（不改变行为）
- 依赖版本升级（常规维护）
- 文档更新

## 创建 ADR 流程

### 1. 复制模板
```bash
cp docs/decisions/ADR-TEMPLATE.md docs/decisions/ADR-006-your-decision.md
```

### 2. 填写内容

**必填章节**:
- 背景（问题是什么？）
- 决策（选择了什么？）
- 考虑的方案（包括被拒绝的）
- 决策依据（为什么选这个？）

**重要章节**:
- 证据（用户确认、原型链接）
- 后果（积极/消极影响）
- 实施（commit、文件路径）

### 3. 关联原型

如果有 UI/UX 原型：
```markdown
## 证据

- **用户确认**: Session 2, 2026-03-11 14:30
- **原型文件**: `docs/prototypes/ui/layout-v2.html` ✅
- **实施 commit**: `dcc3075`
```

### 4. 更新索引

在本 README 的"当前决策记录"表格中添加新行。

### 5. 关联文档

在相关文档中引用 ADR：
- 设计文档: "布局设计见 ADR-001"
- Handoff 文档: "关键决策：ADR-001, ADR-005"

## 使用 ADR 的最佳实践

### 开发前必读

**启动新会话时**:
```markdown
1. [ ] 读取最新 Handoff 文档
2. [ ] 读取关联的 ADR 文档
3. [ ] 读取 ADR 中引用的原型
4. [ ] 理解决策依据
5. [ ] 开始实施
```

### 实施时参考

**开发功能时**:
- 检查是否有关联 ADR
- 如果 ADR 引用了原型，必须查看原型
- 实现应与 ADR 和原型一致
- 如需偏离，先与用户确认并更新 ADR

### 代码审查时检查

**审查 Pull Request 时**:
- 是否有新的决策？如果有，是否创建了 ADR？
- 实现是否符合相关 ADR 的要求？
- 如果修改了决策，ADR 是否更新？

## ADR 状态说明

### 状态定义

| 状态 | 说明 | 何时使用 |
|------|------|---------|
| ⏳ 提议 | 等待讨论和确认 | ADR 刚创建 |
| ✅ 已接受 | 决策已确认，开始实施 | 用户/团队确认 |
| 🔄 实施中 | 正在实施 | 开发进行中 |
| ✅ 已实施 | 实施完成 | commit 已合并 |
| ❌ 已拒绝 | 决策被否决 | 更好的方案出现 |
| 🔮 已取代 | 被新 ADR 替代 | 需求变更 |

### 状态转换

```
⏳ 提议
  ↓ (讨论确认)
✅ 已接受
  ↓ (开始开发)
🔄 实施中
  ↓ (开发完成)
✅ 已实施

或者:
⏳ 提议
  ↓ (更好的方案)
❌ 已拒绝

或者:
✅ 已实施
  ↓ (需求变更)
🔮 已取代 (见 ADR-XXX)
```

## ADR 命名规范

### 文件命名

格式: `ADR-XXX-short-title.md`
- **XXX**: 3 位数字，从 001 开始
- **short-title**: 简短的英文标题（小写，连字符分隔）

**示例**:
- ✅ `ADR-001-ui-layout-two-column.md`
- ✅ `ADR-002-tauri-command-naming-convention.md`
- ❌ `ADR-1-layout.md` (数字不足 3 位)
- ❌ `ADR-002-UI布局.md` (使用中文)

### 标题命名

格式: `ADR-XXX: [简短标题]`
- 使用中文或英文（根据团队习惯）
- 清晰表达决策内容
- 长度控制在 50 字符内

**示例**:
- ✅ "ADR-001: 小说模块使用 2 栏布局"
- ✅ "ADR-002: Tauri Command 参数使用 camelCase"
- ❌ "ADR-003: 权限" (太简略)

## 维护 ADR

### 定期审查

**时机**:
- 每个 Milestone 完成时
- 重大需求变更时
- 季度回顾时

**检查内容**:
- ADR 状态是否准确？
- 是否有未记录的决策？
- 是否有已废弃但未标记的 ADR？

### 更新已有 ADR

**何时更新**:
- 决策部分实施（更新实施 commit）
- 发现新的后果（补充后果章节）
- 状态变更（如"已实施" → "已取代"）

**如何更新**:
1. 不删除原有内容
2. 在"历史"章节添加变更记录
3. 更新状态和相关章节
4. 如果被取代，添加链接到新 ADR

### 废弃 ADR

❌ **不要删除 ADR**（即使已废弃）
- 保留历史决策依据
- 避免重复讨论
- 标记状态: `🔮 已取代 (见 ADR-XXX)`

## 故障排除

### 问题：不知道是否需要创建 ADR

**判断标准**:
- 这个决策重要吗？（影响架构、UX、性能）
- 未来可能忘记为什么这样做吗？
- 其他开发者可能困惑吗？

**如果答案都是"是" → 创建 ADR**

### 问题：ADR 写得太长

**解决**:
- 使用模板的必填章节即可
- "背景""决策""依据"是重点
- 详细技术细节可链接到代码或文档

### 问题：找不到历史决策依据

**解决**:
1. 回溯 git log（commit 消息）
2. 检查聊天记录（需求澄清）
3. 询问用户（如果仍不确定）
4. 标记"依据缺失"并尽可能补充

### 问题：ADR 与实际实现不一致

**原因**:
- ADR 未及时更新
- 实施时偏离了决策

**解决**:
1. 确认当前实现
2. 如果实现正确，更新 ADR
3. 如果实现错误，修正代码
4. 记录差异原因到"历史"章节
